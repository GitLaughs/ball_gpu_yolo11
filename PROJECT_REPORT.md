# AI篮球投篮分析系统项目结题报告

## 1. 项目概述

### 1.1 项目背景

本项目面向篮球训练辅助与智能视频分析场景，目标是构建一套能够对投篮过程进行自动识别、轨迹分析和命中判定的视觉系统。系统以双目拼接视频为输入，结合深度学习目标检测、双目立体视觉、时序轨迹分析与 Web 可视化展示，实现“从视频采集到结果输出”的完整闭环。

### 1.2 建设目标

项目的主要建设目标包括：

1. 自动检测视频中的篮球与篮筐目标。
2. 对检测结果进行跨帧关联，形成稳定轨迹。
3. 利用双目视差恢复目标的空间深度信息。
4. 从篮球轨迹中自动识别投篮出手、飞行与落筐过程。
5. 对投篮结果进行命中/未中判定，并输出命中率统计。
6. 提供浏览器端交互界面，支持上传视频、启动分析、实时查看过程与结果。
7. 提供 Windows 一键安装与启动能力，便于项目交付和验收演示。

### 1.3 最终成果

当前项目已经形成一套可直接运行的工程系统，主要成果如下：

1. 一个基于 Flask 的后端服务程序。
2. 一套完整的目标检测、轨迹构建、物理分析和命中判定算法实现。
3. 一个基于 HTML/CSS/JavaScript 的实时可视化前端页面。
4. 一套适用于 Windows 的 `install_gpu.bat` 与 `start.bat` 自动化运行脚本。
5. 一份可用于项目结题、验收与答辩说明的完整技术报告。

## 3. 项目完整实现流程

## 3.1 部署与启动流程

项目采用“克隆后双击运行”的交付思路，部署链路如下：

### 3.1.1 安装流程

`install_gpu.bat` 负责完成以下任务：

1. 检查 Python 是否存在，优先尝试 `python`，失败时尝试 `py -3`。
2. 检查 NVIDIA GPU 与驱动环境。
3. 创建或复用 `venv` / `.venv` 虚拟环境。
4. 升级 `pip`、`setuptools`、`wheel`。
5. 安装 CUDA 12.1 版 PyTorch。
6. 安装 Flask、OpenCV、NumPy、Ultralytics 等运行依赖。
7. 自动创建 `static`、`uploads`、`exports`、`logs` 目录。
8. 验证 Flask、OpenCV、PyTorch、YOLO 是否可正常导入。

脚本中使用 `chcp 65001` 显式切换为 UTF-8 代码页，以降低 Windows 批处理中文乱码概率。

### 3.1.2 启动流程

`start.bat` 负责：

1. 自动检测虚拟环境。
2. 若未安装则自动调用 `install_gpu.bat`。
3. 检查前端首页与模型文件是否存在。
4. 检查核心依赖是否可导入。
5. 打开浏览器访问 `http://127.0.0.1:5000`。
6. 使用虚拟环境中的 Python 启动 `app.py`。

因此，从交付视角看，使用者通常只需要：

1. 双击 `install_gpu.bat`
2. 双击 `start.bat`

即可完成系统部署与运行。

## 3.2 应用启动后的主控制流程

后端主入口位于 `app.py`。程序启动后主要完成以下初始化：

1. 创建 Flask 应用对象，并指定静态目录为 `./static`。
2. 开启跨域支持。
3. 检测当前设备是否支持 CUDA，并决定使用 GPU 还是 CPU。
4. 初始化全局状态对象 `AppState`。
5. 自动选择可用的默认示例视频。
6. 自动选择模型文件，优先使用 `best_yolo11.pt`，否则使用 `best.pt`。

`AppState` 是整个系统的共享状态中心，其中保存：

1. 当前运行状态、暂停状态、预处理状态。
2. 当前视频路径、模型路径和配置对象。
3. 当前帧、总帧数、FPS、处理耗时、GPU 显存等统计信息。
4. 左画面、右画面、深度图三路最新图像缓存。
5. 投篮阶段、出手次数、命中次数、命中率和最近一次投篮结果。

由于后端同时存在 Flask 请求线程和后台分析线程，因此 `AppState` 内部使用 `threading.RLock()` 保证读写安全。

## 3.3 前端到后端的交互流程

前端页面 `static/index.html` 是一个单页可视化界面，其核心工作机制如下：

1. 用户点击“启动”按钮后，请求 `/api/start`。
2. 用户点击“暂停”按钮后，请求 `/api/pause`。
3. 用户点击“停止”按钮后，请求 `/api/stop`。
4. 用户上传视频时，前端使用 `FormData` 向 `/api/upload_video` 提交文件。
5. 页面通过 `<img>` 标签加载三路 MJPEG 视频流：
   - `/api/video_feed`
   - `/api/video_feed/right`
   - `/api/video_feed/depth`
6. 页面使用定时轮询方式访问 `/api/status`，刷新当前阶段、帧号、FPS、命中率等状态。
7. 页面在左侧主画面上额外叠加 `<canvas>`，用于持续绘制投篮轨迹与出手点。

当前前端还有一个实现细节：后端只返回 `last_shot`，而历史投篮记录 `shotHistory` 由前端根据每次新的 `last_shot` 自行维护。这种设计降低了后端状态复杂度，也使前端渲染更灵活。

## 3.4 后端核心执行流程

系统真正的分析主流程位于 `app.py` 的 `detection_loop()` 中，完整顺序如下：

1. 加载 YOLO 模型。
2. 根据标定参数构建双目矫正对象 `StereoParams`。
3. 创建预处理流水线 `PreprocessPipeline`。
4. 启动预处理后台线程。
5. 等待获取视频基础信息，例如总帧数和帧率。
6. 当前版本为了保证回放阶段画面更平滑，会继续等待预处理全部完成。
7. 创建 `VisionEngine` 和 `ShotEngine`。
8. 重新打开视频，按原始 FPS 进行节流回放。
9. 对每一帧：
   - 读取预处理后的稳定检测结果
   - 调用双目视觉引擎生成右路画面、深度图和 3D 检测信息
   - 选取本帧最可信的篮球与篮筐检测
   - 将投篮出手标记 `is_release_point` 传给投篮状态机
   - 得到投篮阶段与命中事件
   - 调用 `_draw_clean_visualization()` 绘制主画面
   - 更新全局状态供前端读取
10. 视频结束或用户停止后，释放资源并关闭后台线程池。

整个流程可概括为：

```text
用户启动
  -> 加载模型与标定参数
  -> 后台预处理
  -> 回放阶段逐帧分析
  -> 更新视频流与统计
  -> 前端展示结果
```

## 4. 预处理流水线实现

预处理流水线位于 `engine/preprocessor.py`，由 `VideoPreprocessor` 和 `PreprocessPipeline` 两部分组成。

### 4.1 设计目标

预处理流水线的主要目标是把原本“逐帧抖动、不稳定的目标检测结果”，变成“适合进行投篮轨迹分析的稳定时序数据”。当前实现共分为五个阶段。

### 4.2 Phase 1：隔帧批量 YOLO 检测

实现函数：

1. `_parse_yolo_result()`
2. `_phase1_detect_all()`

处理逻辑：

1. 使用 OpenCV 按帧读取视频。
2. 每一帧只截取左半部分图像作为检测输入，因为视频是左右拼接的双目格式。
3. 当前实现只对奇数帧做 YOLO 检测，偶数帧先置为空结果。
4. 多帧累积到 `detect_batch_size` 后调用一次 `model.predict()` 做批量推理。
5. 将 YOLO 输出封装为 `RawDetection2D`。
6. 通过 `frame_result_cb` 把原始检测结果逐批发布到流水线缓存。

这种设计兼顾了三个目标：

1. 使用批量推理提高 GPU 利用率。
2. 通过隔帧检测减少计算量。
3. 为后续轨迹平滑和插值提供基础数据。

### 4.3 Phase 2：跨帧目标跟踪关联

实现函数：

1. `_phase2_build_tracks()`

处理逻辑：

1. 遍历每一帧的原始检测结果。
2. 只在同类别之间进行匹配。
3. 依据 IoU 和中心点距离计算匹配分数。
4. 对所有候选匹配按得分排序，采用贪心策略完成关联。
5. 对未匹配检测新建轨迹。
6. 对长期丢失的轨迹进行结束。
7. 过滤长度过短的轨迹，去除噪声。

输出结果从逐帧检测，变成了“按目标组织的轨迹集合”。

### 4.4 Phase 3：轨迹平滑与缺失插值

实现函数：

1. `_smooth_single_track()`
2. `_phase3_smooth()`

处理逻辑：

1. 对检测框坐标做中位数滤波，抑制离群点。
2. 对宽度、高度做滑动平均，减少尺寸抖动。
3. 对中心点做滑动平均，减少轨迹抖动。
4. 对连续小缺口做线性插值，补齐漏检帧。
5. 插值点使用较低置信度，并打上 `is_interpolated=True` 标记。

这一阶段的作用是把粗糙的目标轨迹修整为更加连续和平滑的轨迹，为物理分析和命中判定提供稳定输入。

### 4.5 Phase 5：物理分析与出手帧检测

当前实现顺序上先做 Phase 5，再做最终导出，原因是需要先确定哪些帧是出手帧，再把这个标记写回到最终检测结果中。

实现函数：

1. `_estimate_pixel_gravity()`
2. `_find_free_flight_segments()`
3. `_find_release_frame()`
4. `_phase5_physics_analysis()`

处理逻辑分为三步：

#### 4.5.1 像素重力估算

1. 选取单条篮球轨迹中的时间窗口。
2. 对纵向坐标进行二次曲线拟合。
3. 由二次项估算像素坐标系中的“重力加速度”。
4. 对多条轨迹的结果取中位数，得到全局像素重力估计值。

#### 4.5.2 自由飞行段识别

1. 在轨迹中滑动窗口。
2. 分别对横向和纵向坐标做多项式拟合。
3. 要求横向加速度足够小，说明主要是抛射运动而不是大范围横向突变。
4. 要求纵向拟合的 `R²` 足够高，说明轨迹近似抛物线。
5. 排除离篮筐过近的区域，避免把碰框、擦网等局部复杂运动误认为标准自由飞行。

#### 4.5.3 出手帧定位

在自由飞行段内部，继续寻找合理的出手点，筛选规则包括：

1. 必须位于画面较下方区域。
2. 必须不是篮筐附近的点。
3. 必须不是明显由弹地反弹造成的上升段。
4. 后续必须存在足够的上升高度。

最终，物理分析结果会生成 `release_frames` 字典，并通过 `PhysicsMetadata` 保存。

### 4.6 Phase 4：最终稳定检测导出

实现函数：

1. `_phase4_export()`

处理逻辑：

1. 将平滑后的轨迹重新展开为逐帧结构。
2. 生成 `StableDetection2D` 对象。
3. 给出手帧写入 `is_release_point=True` 标记。
4. 同一帧同一类别出现多个候选时，优先保留带出手标记或置信度更高的目标。

最终结果会被缓存到 `PreprocessPipeline._stable_dets` 中，供回放阶段按帧读取。

### 4.7 PreprocessPipeline 的工程化实现

`PreprocessPipeline` 是对 `VideoPreprocessor` 的线程化封装，主要用途是：

1. 在后台线程运行预处理。
2. 提前发布 Phase 1 的原始检测结果。
3. 在 Phase 2-5 完成后原子替换为平滑后的最终结果。
4. 对外提供 `wait_for_info()`、`get_dets()`、`frames_ready()` 等方法。

从设计上看，它支持“边预处理边回放”；但当前主程序为了获得更稳定的演示效果，选择等待全量预处理完成后再进入回放阶段。也就是说，流水线能力已经预留，但当前运行策略更偏向“稳定优先”。

## 5. 双目视觉与深度恢复实现

双目相关逻辑位于 `app.py` 的 `build_stereo()` 和 `engine/vision_engine.py`。

### 5.1 标定参数构建

`build_stereo()` 中直接写入了：

1. 左右相机内参矩阵
2. 左右相机畸变系数
3. 双目旋转矩阵 `R`
4. 双目平移向量 `T`

随后调用：

1. `cv2.stereoRectify()`
2. `cv2.initUndistortRectifyMap()`

得到用于双目校正的映射参数。

### 5.2 VisionEngine 的核心处理链路

`VisionEngine.process_frame()` 的处理顺序为：

1. 将拼接帧拆分为左图和右图。
2. 读取上一帧后台立体计算的结果。
3. 将当前帧灰度图提交到后台线程池计算视差。
4. 对已有缓存结果做 2D 到 3D 坐标映射。
5. 返回：
   - 右路画面
   - 深度伪彩图
   - 3D 检测结果 `FrameDetections`

### 5.3 流水线并行设计

当前版本的立体视觉模块使用了单线程 `ThreadPoolExecutor` 作为后台立体匹配线程，原因如下：

1. SGBM 计算较重，适合放到后台执行。
2. OpenCV 的底层实现以 C++ 为主，能够绕开 Python GIL 的部分限制。
3. 主线程不必等待当前帧立体匹配完成，而是使用上一帧缓存结果，从而形成“前后台流水线并行”。

这套设计的直接收益是：

1. 回放线程更稳定。
2. 主线程的显示逻辑与深度计算解耦。
3. 实时展示时的吞吐量更高。

### 5.4 3D 坐标恢复

对于每个稳定检测目标，当前实现不是直接取中心像素的 3D 值，而是：

1. 在中心点周围取一个窗口。
2. 去除无效和全零点。
3. 对剩余点取中位数。

这样可以有效抑制视差噪声，使 3D 坐标更稳定。

需要说明的是：当前主显示画面最终由 `app.py` 中的 `_draw_clean_visualization()` 负责渲染，而 `VisionEngine` 更多承担深度计算、右画面/深度图输出和 3D 信息恢复的职责。

## 6. 投篮状态机与命中判定实现

投篮判定逻辑位于 `engine/shot_engine.py`。

### 6.1 状态机设计

当前版本定义了四个阶段：

1. `IDLE`：等待一次新的投篮启动。
2. `RISING`：篮球被识别为进入上升阶段。
3. `TRACKING`：跟踪篮球接近篮筐后的运动并判断是否命中。
4. `COOLDOWN`：一次投篮结束后的冷却阶段，防止重复计数。

### 6.2 状态机触发逻辑

#### 6.2.1 IDLE 阶段

在 `IDLE` 中，系统不会因为任意篮球出现就直接认为发生了投篮，而是要求：

1. 当前帧被物理分析模块标记为出手帧 `is_physics_release=True`。
2. 该点不能位于篮筐附近。

满足以上条件后，状态机会转入 `RISING`。

#### 6.2.2 RISING 阶段

在 `RISING` 中，系统持续追加轨迹点，并记录当前弧顶位置。当前版本从代码上看，进入 `TRACKING` 的触发条件主要包括：

1. 轨迹长度超过阈值。
2. 篮球已经接近篮筐。
3. 预留的下降计数字段达到阈值。

其中，下降计数变量 `_descending_count` 已在结构上预留，便于后续继续增强“越过弧顶”的判断精度。

#### 6.2.3 TRACKING 阶段

在 `TRACKING` 中，系统重点判断：

1. 球是否进入了篮筐附近区域。
2. 球是否发生了方向反复变化，作为碰框/反弹的依据。
3. 球是否已经明显落到篮筐下方。
4. 轨迹是否超时或过长。

一旦达到结束条件，调用 `_finalize_shot()` 生成完整的投篮事件。

### 6.3 命中判定算法

命中判定由 `_check_scoring()` 完成，当前实现包含三层判断。

#### 第一层：侧边滑出判定

系统首先会检查篮球在接近篮筐区域时是否明显从篮筐左右侧滑出。如果出现明显侧滑，则直接判定为未中，防止假进球。

#### 第二层：绝对穿透判定

如果篮球轨迹先经过篮圈上沿附近，再进入篮网下方区域，并且在这段轨迹内部没有明显偏出，则可直接判定为命中。这一层主要用于修复“空心球减速后不易确认”的问题。

#### 第三层：上下穿越推理

若前两层不能直接给出结论，则继续分析：

1. 轨迹穿越篮圈平面的次数。
2. 轨迹最后的运动方向。
3. 轨迹尾部是否保持在篮圈下方附近。

综合这些信息后，完成最终判定。

### 6.4 结果生成

一旦投篮结束，`_finalize_shot()` 会创建 `ShotEvent`，其中保存：

1. 命中结果
2. 完整轨迹
3. 出手点
4. 弧顶点
5. 结束点
6. 篮筐位置
7. 弧高
8. 持续时间
9. 起止帧号

随后更新总出手数、总命中数，并进入冷却阶段。

## 7. 数据模型设计

系统的数据结构统一定义在 `engine/models.py` 中，核心对象如下：

### 7.1 `RawDetection2D`

表示 Phase 1 的原始检测结果，包含：

1. 类别
2. 置信度
3. 检测框
4. 中心点

### 7.2 `StableDetection2D`

表示平滑后的稳定检测结果，新增：

1. `is_interpolated`
2. `is_release_point`

分别表示该点是否由插值生成、是否被识别为出手点。

### 7.3 `Detection3D`

在二维检测结果基础上进一步加入：

1. 三维坐标 `xyz`
2. 距离 `distance`

便于扩展空间分析功能。

### 7.4 `PhysicsMetadata`

用于保存预处理阶段的全局物理分析信息，包括：

1. 估算得到的像素重力
2. 各条轨迹对应的出手帧

### 7.5 `ShotEvent`

用于描述一次完整的投篮事件，是后续展示与统计的核心数据结构。

## 8. 前端页面实现思路

前端 `static/index.html` 采用单文件实现，但内部结构已经比较清晰。

### 8.1 页面布局

页面主要分为：

1. 顶部状态栏
2. 控制按钮区
3. 左侧主视频区
4. 右侧统计信息区
5. 预处理遮罩层

整体风格采用较强科技感的“赛博球场”视觉设计，便于项目演示和答辩展示。

### 8.2 视频显示方案

当前采用 MJPEG 的方式展示视频流，优点是：

1. 前端实现简单。
2. 浏览器兼容性好。
3. 无需引入额外的视频编解码或 WebRTC 方案。

### 8.3 轨迹绘制方案

前端除了直接显示左路视频，还在视频上方叠加 `<canvas>` 用于绘制轨迹。这样做的优点是：

1. 轨迹绘制与视频解耦。
2. 轨迹可以在 `cooldown` 阶段继续保持显示。
3. 前端可单独控制轨迹样式、颜色、发光效果和释放点标记。

### 8.4 状态轮询方案

系统没有使用 WebSocket，而是采用定时轮询 `/api/status`。这是基于以下考虑：

1. 当前状态量较小，轮询已经足够。
2. 实现简单，稳定性高。
3. 更适合课程设计和项目交付中的快速部署场景。

## 9. 内部代码实现思路总结

从整体实现角度看，本项目采取了以下几个关键设计思路。

### 9.1 先稳后快：预处理优先于实时判定

与“每读一帧立刻做一次完整判定”的方案相比，本项目更强调轨迹质量，因此先通过预处理得到稳定目标，再进入回放分析。这样能够显著降低抖动、漏检和误判对投篮识别的影响。

### 9.2 用物理分析驱动投篮起点识别

本项目没有简单地把“球向上移动”直接等同于投篮，而是通过抛物线拟合、自由飞行段识别和出手帧检测，让投篮状态机的启动更可靠。这使系统更能区分投篮、弹地反弹和随机运动。

### 9.3 用状态机处理时序问题

投篮是一个明显的时序过程，因此采用状态机能够更自然地描述“起手、上升、接近篮筐、结束、冷却”这些阶段，也更利于后续调参与维护。

### 9.4 用后台线程处理重计算模块

预处理和双目立体计算都属于重计算模块，因此项目将预处理整体放入后台线程，将 SGBM 视差计算放入专门的后台线程池，从而降低主流程阻塞。

### 9.5 保持模块职责清晰

当前代码的职责划分较为明确：

1. `app.py` 负责总控和接口。
2. `preprocessor.py` 负责离线预处理与物理分析。
3. `vision_engine.py` 负责双目深度恢复。
4. `shot_engine.py` 负责投篮识别和判定。
5. `index.html` 负责展示与交互。

这种结构便于后续继续演进，例如替换检测模型、增加导出功能或改用更复杂的状态同步机制。

## 10. 当前特点、优势与不足

### 10.1 项目优势

1. 完成了从视频输入到结果展示的完整工程闭环。
2. 同时融合了深度学习、双目视觉、时序状态机与前端展示。
3. 代码模块边界清晰，具备较好的可维护性。
4. 具备实际可运行的交付形态，适合结题演示。
5. 已经考虑了安装脚本、默认资源、前端展示、状态管理等工程细节。

### 10.2 当前不足

1. 当前主流程虽然实现了预处理流水线，但仍选择等待全量预处理结束后再回放，实时性优先级不高。
2. 双目标定参数目前写死在代码中，移植到新设备时需要重新修改代码。
3. 前端历史投篮记录保存在浏览器侧，后端没有持久化保存。
4. 当前规则主要针对现有视频视角和场景调优，换场景后可能需要重新调整阈值。
5. 当前 3D 检测结果尚未深入参与命中判定，未来仍有拓展空间。

### 10.3 后续优化方向

1. 真正启用“边预处理边回放”的流水线模式。
2. 将标定参数外置为配置文件。
3. 增加结果导出、历史存档和日志落盘能力。
4. 将 3D 信息进一步引入投篮轨迹与角度分析。
5. 将前端轮询升级为 WebSocket，减少无效请求。
